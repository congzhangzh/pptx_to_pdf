# PPTX to PDF 服务改造报告

## 背景

项目存在两个版本：

- **`foolish_old_service.py`**：早期可用版本，实现了 PPTX → PDF → OCR 全链路，但有多处设计缺陷
- **`main.py`**：重构后的技术原型，修复了 COM 线程安全问题，但功能不完整（无内存监控、无 OCR 转发）

本次目标：在原型上验证内存监控机制，再整改旧服务输出生产就绪版本。

---

## 旧服务的问题

| 问题 | 根因 |
|---|---|
| PowerPoint 频繁创建/销毁 | 每次请求都调 `Dispatch` + `Quit`，启动开销大，内存无法复用 |
| COM 线程不安全 | `CoInitialize` 在 async 函数里调用，async 函数可能跑在不同线程 |
| OCR URL 硬编码 | `PARSE_API_URL = "http://yyyy/ocr/"` 写死在代码里 |
| 同步阻塞 HTTP | 用 `requests.post` 转发 OCR，阻塞事件循环 |
| 无内存监控 | 长时间运行后内存积累无感知，只能靠外部重启 |

---

## 改造内容

### 阶段一：`main.py` 验证内存机制

**内存监控设计**

同时追踪两个进程的 RSS：
- Python 服务进程本身
- PowerPoint（`POWERPNT.EXE`）— COM 对象持有大量内存，不跟踪的话会漏掉主要来源

```
total = Python RSS + POWERPNT.EXE RSS
```

触发机制：每次转换完成后，通过 FastAPI 的 `BackgroundTask` 在响应发出之后再检查（不阻塞请求返回），超限则对自身发 `SIGTERM`。

关键点：用 `SIGTERM` 而不是 `sys.exit()`，让 uvicorn 有机会走优雅关闭流程（关闭 PowerPoint COM 对象）。

**环境变量**

```
MEMORY_LIMIT_MB=2048   # 默认 2GB，可调低做测试
```

**supervisor.cmd**

外部 bat loop，不嵌入 Python 进程，职责分离：

```bat
:loop
uv run python main.py
timeout /t 2
goto loop
```

服务主动退出（内存超限）→ supervisor 重启 → 进程内存归零。

### 阶段二：`service.py`（旧服务完整重写）

继承 `main.py` 的 `ComWorker` 架构，补上 OCR 转发：

```
PPTX → [COM Worker 专用线程] → PDF → [httpx async] → OCR → JSON
```

- `OCR_URL` 从环境变量读取（默认指向本地 fake_ocr）
- `httpx.AsyncClient` 替换 `requests`，不阻塞事件循环
- 内存监控逻辑同阶段一

### `fake_ocr.py`（测试桩）

极简 FastAPI 服务，接收 PDF 文件上传，返回固定 JSON，用于在没有真实 OCR 引擎的环境下联调全链路：

```json
{"status": "ok", "filename": "demo.pdf", "pages": 1, "text": "fake ocr text"}
```

## 验证结果

```
# 内存触发测试（Linux 环境，无 POWERPNT）
$ MEMORY_LIMIT_MB=1 python -c "_check_memory_and_exit()"
memory 12.4MB / limit 1MB
LIMIT EXCEEDED -> SIGTERM
exit=143   # 128 + 15(SIGTERM) ✓

# fake_ocr 接口
$ curl http://localhost:9000/health
{"status":"ok"}

$ curl -X POST http://localhost:9000/ocr -F "file=@demo.pdf"
{"status":"ok","filename":"demo.pdf","pages":1,"text":"fake ocr text — replace with real OCR engine"}
```
